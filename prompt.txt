Supabase database schema (no SQL, just specification)
General rules - Use Postgres types such as uuid, text, numeric, boolean, timestamptz, date, and time. - Every user who logs in has a row in the profiles table linked to Supabase auth.users by id. - Row Level Security (RLS) is enabled on all user‑facing tables. - Admin users can see and manage all data. - Employee users can only see their own data and the minimum necessary operational information.
Table: profiles Purpose: one row per authenticated user, used for roles and basic identity. Columns: - id: uuid, primary key, same value as auth.users.id. - email: text, unique when present. - full_name: text. - role: text, allowed values: ‘admin’ or ‘employee’. - status: text, allowed values: ‘active’ or ‘inactive’. - created_at: timestamptz, default now. - updated_at: timestamptz, updated on change. RLS idea: - A logged‑in user can select and update only the row where id = auth.uid(). - Users with role = ‘admin’ can select and update all rows.
Table: employees Purpose: extended staff information linked to a profile. Columns: - id: uuid, primary key, foreign key to profiles.id. - display_name: text (short name shown in UI). - position: text (job title, for example “Floor Staff” or “Manager”). - shift_type: text (for example “full‑time”, “part‑time”, “weekend”). - is_active: boolean. - emergency_contact_name: text. - emergency_contact_phone: text. - hr_notes: text (internal notes, not shown to the employee). - created_at: timestamptz, default now. - updated_at: timestamptz. RLS idea: - Admins can read and write all rows. - An employee can read only the row where id belongs to them.
Table: attendance_logs Purpose: track clock‑in and clock‑out for each employee. Columns: - id: uuid, primary key. - employee_id: uuid, foreign key to employees.id. - clock_in: timestamptz. - clock_out: timestamptz, nullable until the employee clocks out. - total_hours: numeric, computed when clock_out is set. - late_flag: boolean, indicates a late arrival according to a rule. - early_checkout_flag: boolean, indicates an early checkout according to a rule. - created_at: timestamptz, default now. - updated_at: timestamptz. Indexes: - Index on employee_id and clock_in for faster queries by employee and date range. RLS idea: - An employee can insert a new row for themselves on clock‑in and update the open row when clocking out. - An employee can select rows where employee_id belongs to them. - Admins can select all rows and export them.
Table: tasks Purpose: define reusable tasks (recurring or one‑time) that can be assigned to employees. Columns: - id: uuid, primary key. - title: text. - description: text. - is_recurring: boolean. - frequency: text (for example ‘daily’ or ‘once’). - location: text (for example ‘main floor’, ‘party room A’). - role_target: text (for example ‘all’, ‘employee’, or a specific role name). - cutoff_time: time, meaning the time of day after which the task is considered overdue. - requires_photo: boolean. - requires_video: boolean. - requires_notes: boolean. - created_by: uuid, foreign key to profiles.id. - created_at: timestamptz, default now. - updated_at: timestamptz. RLS idea: - Admins have full read and write access. - Employees have read‑only access to all task definitions.
Table: task_instances Purpose: represent specific assignments of a task to an employee on a given date. Columns: - id: uuid, primary key. - task_id: uuid, foreign key to tasks.id. - employee_id: uuid, foreign key to employees.id. - scheduled_date: date (the day this instance belongs to). - status: text, allowed values: ‘pending’, ‘completed’, ‘overdue’. - completed_at: timestamptz, nullable until completed. - notes: text. - photo_url: text, nullable, path to the photo in storage. - video_url: text, nullable, path to the video in storage. - created_at: timestamptz, default now. - updated_at: timestamptz. Indexes: - Index on employee_id and scheduled_date. RLS idea: - An employee can select and update rows where employee_id belongs to them. - Admins can select and update all rows.
Table: events Purpose: store parties and events. Columns: - id: uuid, primary key. - title: text. - start_time: timestamptz. - end_time: timestamptz. - room: text. - expected_headcount: integer. - notes: text (for example “parent can leave – extra responsibility”, “extra socks needed”, “discount promised at 6 pm”). - created_by: uuid, foreign key to profiles.id. - created_at: timestamptz, default now. - updated_at: timestamptz. RLS idea: - Admins have full access. - Employees have read‑only access.
Table: event_staff_assignments Purpose: link employees to events with a specific role. Columns: - id: uuid, primary key. - event_id: uuid, foreign key to events.id. - employee_id: uuid, foreign key to employees.id. - role: text (for example ‘host’, ‘cleaning’, ‘check‑in’). - created_at: timestamptz, default now. RLS idea: - Admins can manage all rows. - An employee can see rows where employee_id belongs to them so they know which events they are assigned to.
Table: event_checklists Purpose: track post‑event checklist items. Columns: - id: uuid, primary key. - event_id: uuid, foreign key to events.id. - task_title: text. - status: text, allowed values: ‘pending’ or ‘completed’. - completed_by: uuid, foreign key to employees.id, nullable until completed. - completed_at: timestamptz, nullable until completed. - created_at: timestamptz, default now. RLS idea: - Admins can see and update all rows. - Employees can update rows they are responsible for (for example where completed_by matches them) and can see rows for events where they are assigned.
Table: transactions Purpose: store minimal transactional information needed for staff operations and admin reporting. Columns: - id: uuid, primary key. - event_id: uuid, foreign key to events.id, nullable. - customer_name: text. - phone: text. - amount: numeric. - payment_method: text (for example ‘cash’, ‘card’). - created_at: timestamptz, default now. - handled_by_employee: uuid, foreign key to employees.id. Indexes: - Index on handled_by_employee and created_at. RLS idea: - Admins can select all rows. - For employees, do not expose this table directly. Instead, use a secure view that returns only the most recent five transactions handled by that employee, ordered by created_at descending.
View: employee_recent_transactions (conceptual) Purpose: give employees a safe view of financial data. Behavior: - For the logged‑in user, find their employee_id. - Return at most five rows from transactions where handled_by_employee equals that employee id. - Sort by created_at descending. RLS idea: - The view is readable by authenticated users, but the underlying filter always restricts to their own last five transactions.
Table: waivers Purpose: store metadata about signed waivers. Columns: - id: uuid, primary key. - phone: text. - customer_name: text. - signed_at: timestamptz. - waiver_url: text, link to the waiver document or page. - notes: text. - created_at: timestamptz, default now. RLS idea: - Admins can see all rows. - Employees can search by phone and see basic waiver information but not bulk export everything.
Table: staff_notes Purpose: store quick notes recorded by staff. Columns: - id: uuid, primary key. - event_id: uuid, foreign key to events.id, nullable. - employee_id: uuid, foreign key to employees.id. - note: text. - created_at: timestamptz, default now. RLS idea: - Employees can insert notes with their own employee_id and see their own notes. - Admins can see all notes.
Table: policy_strips Purpose: store short policy and training messages for the staff dashboard. Columns: - id: uuid, primary key. - title: text. - content: text (short sentence or paragraph). - category: text (for example ‘marketing’, ‘liability’, ‘script’). - is_active: boolean. - created_at: timestamptz, default now. - updated_at: timestamptz. RLS idea: - Admins can manage all rows. - Employees can read only rows where is_active is true.
Storage: task media Purpose: store photos and videos linked to task instances. Behavior: - Use a single bucket such as task-media. - Each file path is stored in photo_url or video_url in task_instances. - Employees can only upload and view objects linked to their own task instances. - Admins can access all media. - A background job periodically deletes objects older than 30 days and clears stale URLs in task_instances.
Next.js application structure (no code, just specification)
Overall layout - Use the App Router. - Have a root layout that sets up global styles and a basic shell. - After login, route admins into an /admin section and employees into an /employee section.
Supabase client setup - Create a single Supabase client configuration in a shared file. - Use a browser‑safe client for client side calls and a separate server‑side client for operations that use the service role key. - Never expose the service role key to the browser.
Auth context - Implement a shared auth context or hook that: - Tracks the current Supabase session. - Loads the corresponding profiles row. - Exposes helpers such as isAdmin and isEmployee. - Use this information to protect routes so that only admins see admin pages and only employees see employee pages.
Routing and pages - Public routes: a login page and any simple info pages if needed. - Admin routes under /admin with pages for: - Dashboard (high level metrics and links). - Employees (manage profiles and employees). - Tasks (manage task definitions and see task completion stats). - Events (manage events and staffing). - Attendance (view and export attendance logs). - Reports (attendance and task reports, event issues). - Policies (manage policy_strips rows). - Employee routes under /employee with pages for: - Dashboard (clock in/out, today’s tasks, upcoming events, active policy strips). - Check‑in tools (search waivers by phone, view party details, add staff notes). - Attendance (personal history view only). - Tasks (history of completed tasks if needed).
Feature folders - Organise code by feature: attendance, tasks, events, waivers, policies, users. - In each feature folder, keep pages, reusable UI components, and data access helpers together.
Data access and error handling - Wrap Supabase queries inside small helper functions or hooks per feature. - Always handle loading and error states. - Show friendly error messages in the UI instead of raw errors.
Dashboards - Admin dashboard should show small summary cards for active employees, today’s attendance, task completion rate, and upcoming events, with links to the detailed pages. - Employee dashboard should focus on what an employee needs today: clock in/out, today’s tasks, the next few events they are assigned to, and the current policy messages.
Security and access assumptions - Always assume RLS is the primary security layer. - The frontend should only request the minimum data needed for each view. - Do not attempt to bypass or duplicate security rules in the client.
